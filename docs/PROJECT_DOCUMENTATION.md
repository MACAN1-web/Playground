# Документация приложения «Рейтинг абитуриентов колледжа»

Документ описывает, как устроено приложение, как его запускать, как обновлять рейтинги, где менять настройки и какие места важно не сломать при доработках.

## 1. Назначение приложения

Приложение нужно для публикации рейтингов абитуриентов колледжа на сайте.

Основные сценарии:

- абитуриент открывает публичную страницу и ищет себя по полному СНИЛС;
- абитуриент смотрит опубликованные списки по специальностям;
- сотрудник колледжа открывает скрытую админку `/admin`;
- сотрудник загружает Excel-файл с реестром заявлений;
- система распределяет абитуриентов по специальностям;
- сотрудник внутри админки отмечает, кто принёс оригинал документа;
- отметка оригинала не показывается публично, но поднимает абитуриента выше тех, кто оригинал не принёс.

## 2. Технологии

- `React` — интерфейс приложения.
- `TypeScript` — типизация frontend и backend.
- `Vite` — запуск и сборка frontend.
- `Express` — backend API.
- `PostgreSQL` — основная база данных.
- `Docker Compose` — локальный запуск PostgreSQL.
- `xlsx` — чтение Excel-файлов.
- `multer` — загрузка Excel-файла в API.
- `pg` — подключение backend к PostgreSQL.

## 4. Быстрый запуск

Первый запуск:

```bash
cp .env.example .env
npm install
docker compose up -d postgres
npm run db:migrate
npm run dev
```

Обычный запуск после настройки:

```bash
docker compose up -d postgres
npm run dev
```

После запуска Vite покажет адрес:

```text
Local: http://localhost:5173/
```

Если порт занят, адрес может быть `http://localhost:5174/`, `http://localhost:5175/` и так далее.

Backend API работает на:

```text
http://localhost:3001
```


```text

```

## 5. Настройки `.env`

Файл `.env` содержит локальные настройки:

```env
ADMIN_PASSWORD=пароль-админки
AUTH_SECRET=секрет-для-токена
PORT=3001
DATABASE_URL=postgresql://college_app:local-development-password@127.0.0.1:5432/college_rating
APP_ORIGIN=http://localhost:5173
```

### `ADMIN_PASSWORD`

Пароль для входа в админку.

Чтобы изменить пароль:

1. Откройте `.env`.
2. Измените строку `ADMIN_PASSWORD=...`.
3. Перезапустите приложение.

```bash
npm run dev
```

### `AUTH_SECRET`

Секретная строка для подписи админской сессии.

Если изменить `AUTH_SECRET`, все старые админские сессии станут недействительными. Это полезно, если пароль мог попасть не туда.

Сгенерировать новый секрет:

```bash
openssl rand -hex 32
```

### `DATABASE_URL`

Строка подключения к PostgreSQL.

Локально используется:

```env
DATABASE_URL=postgresql://college_app:local-development-password@127.0.0.1:5432/college_rating
```

На сервере колледжа нужно будет заменить логин, пароль, адрес и название базы на реальные.

### `APP_ORIGIN`

Адрес сайта, с которого браузеру разрешено обращаться к API.

Локально:

```env
APP_ORIGIN=http://localhost:5173
```

На сервере колледжа нужно указать реальный HTTPS-адрес сайта. В production приложение не должно работать с открытым CORS.

## 7. Миграции базы

Структура базы описана в:

```text
server/migrations/001_initial.sql
```

Применить миграцию:

```bash
npm run db:migrate
```

Сейчас миграция создаёт две основные таблицы:

- `directions` — специальности;
- `applicants` — абитуриенты внутри специальностей.

## 8. Таблицы базы данных

### `directions`

Хранит специальности.

Важные поля:

- `id` — внутренний ID специальности;
- `specialty` — название специальности;
- `specialty_search` — строка для поиска по специальности;
- `study_form` — форма обучения;
- `funding` — тип финансирования, сейчас используется как техническое поле;
- `budget_places` — количество бюджетных мест, сейчас публично не показывается;
- `updated_at` — дата и время публикации рейтинга.

### `applicants`

Хранит абитуриентов.

Важные поля:

- `id` — внутренний ID записи;
- `direction_id` — связь со специальностью;
- `position` — место в рейтинге;
- `snils_normalized` — СНИЛС только цифрами;
- `average_score` — средний балл;
- `original_status` — статус заявления из Excel, хранится технически и публично не отображается;
- `full_name` — ФИО для админки;
- `full_name_search` — строка для поиска по ФИО;
- `original_provided` — внутренняя отметка «оригинал предоставлен».

## 9. Формат Excel-файла

Основной формат — единый реестр заявлений.

Обязательные колонки:

- `Специальность`
- `Форма обучения`
- `СНИЛС абитуриента`
- `Средний балл аттестата`
- `Статус заявления` — может быть в файле, но на сайте не отображается

Для ФИО используются колонки:

- `Фамилия абитуриента`
- `Имя абитуриента`
- `Отчество абитуриента`

Логика импорта:

1. Система читает первый лист Excel-файла.
2. Проверяет, что это реестр по колонкам `Специальность` и `СНИЛС абитуриента`.
3. Группирует строки по паре `Специальность + Форма обучения`.
4. Одинаковая специальность на очной и заочной форме создаёт два отдельных списка.
5. Внутри одного списка объединяет дубли по СНИЛС.
6. Если один СНИЛС встречается в разных специальностях или формах обучения, он попадает во все эти списки.
7. Если дубль внутри одного списка имеет разные баллы, остаётся запись с большим баллом.
8. Старые опубликованные рейтинги заменяются новым файлом.
9. Отметки `original_provided` сохраняются после повторной загрузки файла.

Ограничения:

- максимум 50 специальностей в одном файле;
- максимум 100 000 записей;
- файл должен содержать СНИЛС;
- СНИЛС должен содержать 11 цифр.

## 10. Как обновлять рейтинги

1. Запустить приложение.
2. Открыть `/admin`.
3. Ввести пароль из `.env`.
4. В блоке «Быстрая публикация» загрузить `.xls` или `.xlsx`.
5. Дождаться сообщения об успешной публикации.
6. Проверить публичные списки.

При успешной загрузке система покажет:

- сколько специальностей опубликовано;
- сколько абитуриентов импортировано;
- сколько дублей объединено;
- сколько строк пропущено.

Если файл ошибочный, старый рейтинг не должен замениться.

## 11. Публичная часть

Публичная часть доступна всем пользователям.

Есть две вкладки:

### «Найти себя»

Абитуриент вводит полный СНИЛС из 11 цифр.

Если СНИЛС найден, система показывает:

- специальность;
- место;
- средний балл;
- дату обновления.

СНИЛС отображается маской:

```text
123*****789
```

### «Списки поступающих»

Пользователь выбирает специальность и видит список:

- место;
- маскированный СНИЛС;
- средний балл.

ФИО публично не показывается.

Отметка оригинала публично не показывается.

## 12. Админка

Админка доступна по адресу:

```text
/admin
```

Возможности:

- загрузить новый Excel-файл;
- искать абитуриентов по ФИО;
- искать по полному СНИЛС;
- искать по части названия специальности;
- искать по коду специальности;
- раскрыть специальность и посмотреть внутренний список с ФИО;
- отметить галочкой, что абитуриент принёс оригинал;
- очистить рейтинг специальности;
- удалить специальность.

## 13. Оригиналы документов

Отметка `Оригинал предоставлен` работает только внутри админки и только для выбранной специальности.

Правила:

- у одного СНИЛС может быть только одна активная галочка оригинала;
- если сотрудник ставит галочку в другой специальности, старая галочка этого СНИЛС снимается автоматически;
- галочка сохраняется после новой загрузки Excel-файла;
- публично галочка не отображается;
- абитуриенты с оригиналом поднимаются выше всех абитуриентов без оригинала.

Сортировка рейтинга:

1. Сначала все абитуриенты с `original_provided = true`.
2. Внутри группы с оригиналом — больший средний балл.
3. Затем абитуриенты без оригинала.
4. Внутри группы без оригинала — больший средний балл.
5. Если и это одинаково — сортировка по СНИЛС.

## 14. Защита персональных данных

Что сделано для снижения риска:

- публично СНИЛС отображается только маской;
- публично ФИО не показывается;
- публичный поиск работает только по полному СНИЛС;
- ФИО доступно только в админке;
- отметка оригинала доступна только в админке;
- паспортные и контактные данные не сохраняются.

Важно: финальное решение по соответствию 152-ФЗ должен подтвердить ответственный за персональные данные в колледже.

## 15. API

### Публичные endpoints

#### `GET /api/directions`

Возвращает список специальностей.

#### `GET /api/directions/:id/applicants`

Возвращает публичный список абитуриентов по специальности.

СНИЛС маскируется.

#### `POST /api/search`

Поиск по полному СНИЛС.

Пример тела запроса:

```json
{
  "snils": "12345678901"
}
```

### Админские endpoints

Для админских endpoints нужен заголовок:

```text
Authorization: Bearer <token>
```

#### `POST /api/admin/login`

Вход в админку.

```json
{
  "password": "пароль"
}
```

#### `POST /api/admin/import-workbook`

Загрузка Excel-файла.

Используется `multipart/form-data`, поле файла называется `file`.

#### `GET /api/admin/applicants?q=...`

Внутренний поиск абитуриентов.

Работает по:

- ФИО;
- полному СНИЛС;
- названию специальности;
- коду специальности.

#### `GET /api/admin/directions/:id/applicants`

Внутренний список абитуриентов по специальности.

Возвращает ФИО и отметку оригинала.

#### `PATCH /api/admin/applicants/:snils/original`

Изменение отметки оригинала.

```json
{
  "directionId": 1,
  "originalProvided": true
}
```

#### `DELETE /api/admin/directions/:id/rating`

Очищает рейтинг специальности, но оставляет саму специальность.

#### `DELETE /api/admin/directions/:id`

Удаляет специальность вместе с рейтингом.

## 18. Перенос старой SQLite-базы

Скрипт:

```text
scripts/transfer-sqlite-to-postgres.ts
```

По умолчанию читает файл:

```text
data/ratings.db
```

Команда:

```bash
npm run db:transfer-sqlite
```

Если нужно указать другой SQLite-файл:

```bash
SQLITE_PATH=/путь/к/ratings.db npm run db:transfer-sqlite
```

Важно: перенос полностью очищает PostgreSQL-таблицы `applicants` и `directions`, а потом записывает туда данные из SQLite.

## 20. Частые проблемы

### Белый экран

Проверить:

1. Запущен ли frontend.
2. Какой адрес показал Vite.
3. Нет ли ошибок в консоли браузера.
4. Работает ли API.

Проверка API:

```bash
curl http://127.0.0.1:3001/api/directions
```

### Админка пишет «Неверный пароль»

Проверить:

1. Пароль в `.env`.
2. Перезапущен ли `npm run dev` после изменения `.env`.
3. Не висит ли старый процесс на порту `3001`.

Посмотреть процесс:

```bash
lsof -nP -iTCP:3001 -sTCP:LISTEN
```

Если висит старый процесс, его можно остановить:

```bash
kill <PID>
```

### API не подключается к PostgreSQL

Проверить Docker:

```bash
docker compose ps
```

Запустить PostgreSQL:

```bash
docker compose up -d postgres
```

Проверить `DATABASE_URL` в `.env`.

### После загрузки Excel мало абитуриентов

Возможные причины:

- в строках нет СНИЛС;
- СНИЛС не содержит 11 цифр;
- нет среднего балла;
- средний балл не распознался как число;
- строки объединены как дубли внутри одной специальности.

### Специальность не появляется

Проверить колонку:

```text
Специальность
```

Она должна быть заполнена.

## 21. Резервное копирование

Для PostgreSQL лучше использовать `pg_dump`.

Пример для Docker:

```bash
docker compose exec -T postgres pg_dump -U college_app college_rating > backup.sql
```

Восстановление:

```bash
docker compose exec -T postgres psql -U college_app college_rating < backup.sql
```

Перед восстановлением лучше сделать отдельную копию текущей базы.

## 22. Рекомендации для production

Перед размещением на сервере колледжа:

- задать сильный `ADMIN_PASSWORD`;
- задать случайный длинный `AUTH_SECRET`;
- задать `APP_ORIGIN` с HTTPS-адресом сайта колледжа;
- использовать отдельный пароль PostgreSQL;
- не хранить production-пароли в публичном репозитории;
- запустить приложение за HTTPS;
- настроить reverse proxy;
- настроить регулярный backup PostgreSQL;
- ограничить доступ к серверу;
- согласовать публикацию маскированного СНИЛС с ответственным за персональные данные.

## 23. Защитные механизмы

В приложении включены базовые меры защиты:

- CORS ограничивается через `APP_ORIGIN`;
- login ограничен по количеству попыток;
- публичный поиск по СНИЛС ограничен по частоте запросов;
- пароль сравнивается через безопасное сравнение строк;
- загрузка рейтинга принимает только `.xls` и `.xlsx`;
- API отдаёт базовые security headers;
- публичные endpoints не возвращают ФИО, полный СНИЛС и отметку оригинала.

- Telegram-бот после стабилизации web-версии;

## 25. Важные архитектурные решения

### Почему PostgreSQL

PostgreSQL надёжнее для серверного приложения:

- лучше подходит для одновременной работы;
- проще делать backup;
- проще переносить на сервер;
- безопаснее, чем файл SQLite на диске;
- удобнее развивать проект дальше.

### Почему один пароль админки

Сейчас приложение простое, поэтому один общий пароль экономит время.

Минусы такого подхода:

- нельзя узнать, кто именно изменил галочку;
- нельзя дать разные права разным сотрудникам;
- при утечке пароля его нужно менять сразу всем.

Если потребуется контроль действий, лучше добавить пользователей и журнал аудита.

### Почему оригинал скрыт публично

Отметка оригинала используется как внутренний фактор сортировки: все абитуриенты с оригиналом поднимаются выше тех, кто оригинал не принёс. Публично её не показываем, чтобы не создавать лишних вопросов и рисков.

### Почему поиск по ФИО только в админке

ФИО — персональные данные. Публично оно не нужно для задачи поиска себя в рейтинге, поэтому в публичном интерфейсе используется только полный СНИЛС и маскированный вывод.
